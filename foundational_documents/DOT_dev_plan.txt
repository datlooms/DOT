 PHASE 2 — SEQUENTIAL-LATCH SUBSYSTEM (change b)
 The 2 F1 signals only. A bounded lagged check — NOT a second engine.
 Target sections: 3.0 (state), 6.8 (rule table), 6.9 (signal evaluation).
================================================================================

  BACKGROUND. An F1 signal is "leg A fires, then k bars later leg B fires, with
  an ST_Flip event present at the leg-A bar." The EA already computes every
  variable involved and already tracks ST_Flip (ST_Flip_Event / Bars_Since_Flip).
  The only new machinery is a small rolling memory of the leg-A latch.

  THE 2 F1 SIGNALS:
    F1-SB: Sqz_Val:hi ->13-> Micro_OrderFlowDelta:lo  [LONG]
    F1-TE: ADX_Rising:==0 ->8-> D2D_DirStep:==-1       [LONG]
  Shared leg-A families: Sqz_Val (for SB) and ADX_Rising (for TE). The ST_Flip
  anchor is shared by both.

  Step 6.  STATE (SECTION 3.0). Add two fixed-length ring buffers of booleans,
           length 16 (covers lag ≤ 15 with margin), one per F1 leg-A latch:
               bool latchBuf_SqzHiFlip[16];    // (Sqz_Val:hi AND ST_Flip) latch
               bool latchBuf_ADXris0Flip[16];  // (ADX_Rising==0 AND ST_Flip) latch
           plus a single rolling write-index. These are the ONLY new persistent
           state. They resize/shift under ResizeAndSmartShift discipline like all
           per-bar state (one shift per bar; no terminal overrides).

  Step 7.  LATCH WRITE (once per bar, in the DOTS derived/eval pass, SECTION 6.9
           entry point, AFTER the variables and ST_Flip_Event for the bar are
           final). On bar i:
               latchBuf_SqzHiFlip[idx]   = (Sqz_Val >= thr_SqzVal_hi) && (ST_Flip_Event != 0);
               latchBuf_ADXris0Flip[idx] = (ADX_Rising == 0)          && (ST_Flip_Event != 0);
           where thr_SqzVal_hi is the SAME adaptive threshold the F0 path uses
           for Sqz_Val:hi (route through the existing threshold table — do NOT
           introduce a new threshold source). Advance idx = (idx + 1) % 16.

  Step 8.  LAGGED FIRE (same pass, using the CURRENT bar's leg-B conditions).
           For F1-SB (lag 13):
               legA13 = latchBuf_SqzHiFlip[(idx - 13 + 16) % 16];
               legB   = (Micro_OrderFlowDelta <= thr_OFDelta_lo);
               fireSB = legA13 && legB && (D2D_Trend_Dir == +1);
           For F1-TE (lag 8):
               legA8  = latchBuf_ADXris0Flip[(idx - 8 + 16) % 16];
               legB   = (D2D_DirStep == -1);
               fireTE = legA8 && legB && (D2D_Trend_Dir == +1);
           Route thr_OFDelta_lo through the existing adaptive threshold table.
           ADX_Rising==0 and D2D_DirStep==−1 are direct state equalities on
           existing variables (no threshold).

  Step 9.  RULE TABLE (SECTION 6.8, InitDotsRuleTable). Register the 2 F1 signals
           as entries of a new trigger TYPE (a triggerKind flag: F0_TRIPLE vs
           F1_SEQ) so the panel and the signal-evaluation dispatcher treat them
           correctly. Their direction is LONG; their structures SQUEEZE_BREAKOUT
           and TREND_EXHAUSTION. They feed the SAME entry gate (≥2 concurrent OR
           solo+Vol≥300) and the SAME S.7/runner exit as every F0 signal.

  Step 10. NO NEW EXIT, NO NEW GATE, NO SECOND LOOP. Once an F1 signal fires, it
           is an ordinary Dots entry. The subsystem is purely the latch buffers +
           lagged lookups feeding the existing evaluation/dispatch path.

 PHASE 4 — 6-LOT LIVE-RISK JAR (change d)
 Replace the MAX_POSITIONS=6 count cap with a 6-lot LIVE-risk budget.
 It counts RISK, not bodies: break-even'd winners free their lot and no longer block.
 Target sections: 3.0 (state), 8.6 (Dots entry/admission), 8.1 (BE transition).
================================================================================

  BACKGROUND. The old control blocked a 7th position whenever 6 were open — even if
  all 6 were already winning and at break-even (zero further risk). The jar instead
  bounds LIVE (pre-break-even) risk to 6 lots. Every trade is a whole 1 lot; a position
  at break-even carries no risk, so its lot leaves the jar and a new signal may take it.
  Validated (1 lot): 2,409 tr (+74), net $58,685 (+$1,418), PF 5.83, WR 91.9%, worst-day
  -$127.5 and max-DD -$165.6 IDENTICAL to the count cap, 6/6 folds, 22/22 weeks, OOS 6.57.
  Strict improvement at the IDENTICAL 6-lot hard bound.

  Step 15. STATE (SECTION 3.0). Add a single running live-lot counter, e.g.
           int g_dots_live_lots = 0; It holds the number of open Dots positions NOT yet
           at break-even (each = 1 lot). This is the ONLY new persistent state for the jar.

  Step 16. ADMISSION (SECTION 8.6, Dots entry). REPLACE the MAX_POSITIONS count check
           with the jar check: a new Dots signal may open ONLY IF g_dots_live_lots < 6.
           If the jar is full of live risk (== 6), the signal is NOT taken (it is skipped,
           exactly as the count cap skipped). No fractional sizing — always 1 lot.

  Step 17. INCREMENT (SECTION 8.6, on open). When a Dots position opens, g_dots_live_lots
           += 1 (it enters the jar as live risk).

  Step 18. BE-FREE (SECTION 8.1, the BE transition). At the exact bar a Dots position first
           reaches break-even (its SL is moved to entry±step — step 3, it can no longer
           lose), g_dots_live_lots -= 1. This is the BE-freeing mechanic: the winner's lot
           leaves the jar. Decrement EXACTLY ONCE, on the pre-BE -> BE transition (guard on
           the same be flag that gates the one-time BE nudge, so no double-decrement).

  Step 19. EXIT (SECTION 8.6 / 8.1, on close). When a Dots position CLOSES while still LIVE
           (exited before ever reaching break-even — an SL hit), g_dots_live_lots -= 1. A
           position that closes AFTER break-even was ALREADY decremented at Step 18 — do NOT
           decrement again. Net invariant: g_dots_live_lots == count of open, not-yet-BE Dots
           positions, and NEVER exceeds 6.

  DO NOT TOUCH: SL / step / BE nudge / tier / LeapFrog / runner / Friday close / entry gate /
  eligibility — all remain S.7. The jar changes ADMISSION only, keyed on live-lot count.

 PHASE 5 — PER-BAR EXECUTION MODEL (change e)
================================================================================

  Revert the DOTS SL-management path from PER-TICK to PER-BAR so live == the validated
  per-bar book. The EA currently repositions the SL on live ticks; it must reposition ONLY
  on a NEW BAR, off the CLOSED bar's High/Low, reproducing portfolio_simulation_engine.py.

  Step 20. GATE TO NEW-BAR (SECTION 8.6 / 8.1, ManageDotsPositions path). Wrap the SL
           reposition logic (arm / tier / trail) behind new-bar detection so it runs ONCE
           per closed bar, not on every tick. Do NOT gate the broker SL — the hard SL stays
           live and executes intrabar on any tick (downside protection is always per-tick).
  Step 21. USE CLOSED-BAR EXTREMES. Compute favourable-move / tier count / arm / trail off
           the CLOSED bar's High/Low (High[1]/Low[1]), NOT the live Bid tick and NOT Close.
           Long: favourable = High[1] - entry; Short: favourable = entry - Low[1].
  Step 22. PER-BAR SEQUENCE (exact order, matches the sim): on each new bar, per open Dots
           position — (1) exit-check vs current SL, (2) update tiers off the closed bar
           high/low, (3) arm/lock BE (SL -> entry ± be_lock_dist), (4) trail (tier ≥ 3 ->
           entry ± (tiers - lag) × step). This order reproduces the validated book.

  RATIONALE (why per-bar, not per-tick): with LOCK_FRAC = 1.0 the lock lands on the
  tier-1/trigger price. Under PER-TICK the SL sits on price the instant it arms, so the next
  adverse tick scratches it and runners collapse (387 -> 102) before they can run. PER-BAR
  holds the armed stop through the minute's intrabar noise, letting trades reach tier 3+.
  Per-bar IS the validated $58,685 book; per-tick diverges downward only ($33.9k worst-case).
  This is a parity fix — the same discipline as export = live and jar-parity.

  DO NOT TOUCH: LOCK_FRAC = 1.0, BE_TRIG_FRAC = 1.0, the SL/step/BE/tier/trail arithmetic,
  or the broker's hard-SL execution. Change (e) alters WHEN the EA repositions the SL, not
  the values it computes.
